## 📋 Краткое описание проекта

**cudaCrimea** - Telegram-бот агрегатор событий в Крыму (Севастополь, Симферополь, Ялта и др.).

**Цель:** Помочь пользователям найти интересные события (концерты, выставки, спектакли, экскурсии и т.д.) в их городе.

### Технологический стек:

**Backend:**
- Python 3.11+
- FastAPI (REST API)
- aiogram 3.x (Telegram Bot)
- SQLAlchemy 2.x + asyncpg (ORM)
- APScheduler (фоновые задачи)

**База данных:**
- PostgreSQL 15
- Redis 7

**AI/LLM:**
- AI Mediator API (или OpenAI)
- Модели: gpt-4o-mini
- Промпты хранятся в БД

**Frontend (Admin):**
- Jinja2 Templates
- Vanilla JS
- Tailwind CSS (через CDN)

**Инфраструктура:**
- Docker + Docker Compose
- Nginx (reverse proxy)
- Сервер: VPS (5.83.140.80)

## ✅ Реализованные функции

### 1. Telegram Bot

**Основные команды:**
- `/start` - Приветствие и главное меню
- Поиск событий по фильтрам:
  - По дате (сегодня, завтра, выходные, на неделю, на месяц)
  - "🔥 Горячее" - события от текущего времени до 8:00 следующего дня
  - По категориям (концерты, театр, детям, экскурсии, вечеринки, выставки)
  - По бюджету
  - По городу

**Навигация по событиям:**
- Карточки событий с inline-кнопками
- Фото/видео события (если есть)
- Кнопка "Подробнее" с источником
- Переключение между событиями (◀️ ▶️)

**UGC (User Generated Content):**
- Пользователи могут предложить своё событие
- Форма с полями: название, дата, время, место, цена, категория
- Модерация в админ-панели

### 2. REST API

**Эндпоинты для бота:**
- `GET /api/events` - Поиск событий с фильтрами
- `GET /api/events/hot` - Горячие события
- `GET /api/cities` - Список городов
- `POST /api/ugc/submit` - Отправка UGC

**Фильтрация:**
- По городу
- По дате (date, when)
- По категориям
- По бюджету (budget_max)
- По близости (lat, lon, radius) с fallback для событий без координат

**Особенности:**
- Кеширование в Redis (TTL 5 минут)
- Поддержка событий без координат (lat/lon = NULL)
- Горячие события: с текущего момента до 8:00 следующего дня

### 3. Админ-панель

**URL:** `https://ваш-домен.ru/admin`

**Разделы:**

**События (`/admin/events`):**
- Список всех событий с фильтрами
- Фильтры: город, категория, статус (active/past/draft/pending_moderation)
- Редактирование события
- Отображение медиа (фото/видео)
- Статусы видны на отдельной строке под названием

**UGC Модерация (`/admin/ugc`):**
- Очередь событий от пользователей и парсеров
- 3 очереди:
  - `ugc:queue` - от пользователей
  - `ugc:queue:parser` - от парсеров
  - `ugc:queue:bulk` - массовая обработка AI
- Кнопки: Одобрить / Отклонить / Редактировать
- AI-обработка: классификация + извлечение данных
- Bulk actions: массовое удаление, AI-обработка

**Парсеры (`/admin/parsers`):**
- Управление парсерами (включить/выключить)
- Настройка расписания
- Ручной запуск парсера
- Статистика парсинга

**LLM Промпты (`/admin/llm`):**
- Управление промптами для AI
- 2 типа: classifier (определение события), extractor (извлечение данных)
- Активация/деактивация промптов
- История использования токенов

**Статистика (`/admin/stats`):**
- Общая статистика событий
- LLM usage (расход токенов)
- Популярные категории

**Платные размещения (`/admin/placements`):**
- Просмотр заявок на платное размещение
- Модерация и одобрение
- Отслеживание статусов

### 4. Парсинг событий (AI-based)

**Архитектура парсинга:**

**До (старый подход):**
- Жёсткие HTML-селекторы для каждого сайта
- Регулярные выражения для извлечения данных
- Сложная поддержка при изменении разметки

**После (текущий подход - AI):**
- Универсальный AI-парсер (`app/ingestors/ai_parser_base.py`)
- Двухэтапный процесс:
  1. **Классификатор** - определяет, является ли текст событием
  2. **Экстрактор** - извлекает структурированные данные
- Redis-кеширование результатов (TTL 24 часа)
- Автоматическое извлечение контактов (телефон, email, telegram, vk, instagram)

**Мигрированные парсеры:**
- ✅ **KudaGo** (`kudago.py`) - API парсер с AI-обработкой JSON
- ✅ **Yandex Afisha** (`yandex_afisha.py`) - HTML → AI
- ✅ **Afisha Goroda** (`afisha_goroda.py`) - HTML → AI
- ✅ **Kassa24** (`kassa24.py`) - HTML → AI
- ✅ **Telegram Channels** (`tg_channels.py` + `tg_ai_extractor.py`) - уже использовал AI

**Преимущества AI-парсинга:**
- Устойчив к изменениям разметки сайтов
- Универсальный подход для всех источников
- Понимает контекст (не просто ищет селекторы)
- Автоматическая категоризация событий
- Легко добавлять новые источники

**Расписание парсинга:**
- KudaGo: каждые 6 часов
- Yandex Afisha: каждые 4 часа
- Telegram: каждые 45 минут
- Автоархивация прошедших событий: каждые 30 минут

### 5. База данных

**Основные таблицы:**

**events** - События
- Поля: id, title, date, time, city, venue_name, address, lat, lon
- Цена: price_min, price_max, is_free
- Категория: category (concert, theatre, kids, tour, party, expo, sport, other)
- Статус: status (active, past, draft, pending_moderation)
- Медиа: images (JSONB массив URL)
- Источник: source, source_url

**placements** - Платные размещения
- Связь с событием через event_id
- Статусы: pending → approved → paid → active → completed
- Цена, период размещения

**llm_usage** - Использование LLM
- Отслеживание расхода токенов
- service, model, provider
- prompt_tokens, completion_tokens, total_tokens

**llm_prompts** - Промпты для AI
- Типы: classifier, extractor
- Версионирование промптов
- is_active флаг

**telegram_accounts** - Telegram аккаунты для парсинга
- Для доступа к закрытым каналам через Telethon

## 🐛 Известные проблемы и решения

### 1. ✅ Решено: Бот показывал "Сервис временно недоступен"
**Причина:** Неправильный entry point в docker-compose (`app.admin.main:app` вместо `app.api.main:app`)
**Решение:** Исправлен docker-compose.prod.yml

### 2. ✅ Решено: Админ-панель редирект-луп
**Причина:** Nginx rewrite конфликтовал с FastAPI маршрутами
**Решение:** Убран rewrite в nginx.prod.conf

### 3. ✅ Решено: API возвращал 0 событий при поиске
**Причина:** Bbox-фильтр исключал события без координат
**Решение:** Добавлен `OR Event.lat == None` в фильтр

### 4. ✅ Решено: Бот зависал на "Ищу горячие события"
**Причина:** Telegram API ошибка - относительные URL в image_url
**Решение:** Валидация URL (только абсолютные http/https)

### 5. ✅ Решено: Метка статуса не видна при длинном названии
**Причина:** Метка была inline с названием
**Решение:** Перенесена на отдельную строку под названием

## 💡 Важные заметки для контекста

### Когда терять контекст - читай это:

1. **База данных называется `cudacrimea`, не `cuda_db`**
2. **Пользователь PostgreSQL: `postgres`, не `cuda_user`**
3. **SSH ключ для сервера:** `~/.ssh/cudacrimea_key`
4. **Сервер:** root@5.83.140.80
5. **AI парсинг - ЭТО ВАЖНО:** Все парсеры теперь используют LLM, не селекторы!
6. **Промпты в БД:** Хранятся в таблице `llm_prompts`, можно менять без кода
7. **Статусы событий:** active, past, draft, pending_moderation
8. **Горячие события:** От текущего времени до 8:00 следующего дня
9. **UGC модерация:** Три очереди в Redis (ugc:queue, ugc:queue:parser, ugc:queue:bulk)
10. **Метки статусов:** Теперь на отдельной строке под названием (не inline)

### Команды для быстрого подключения к серверу:

```bash
# Проверить контейнеры
ssh -i ~/.ssh/cudacrimea_key root@5.83.140.80 "docker ps"

# Подключиться к БД
ssh -i ~/.ssh/cudacrimea_key root@5.83.140.80 "docker exec cuda_db psql -U postgres -d cudacrimea"

# Очистить Redis
ssh -i ~/.ssh/cudacrimea_key root@5.83.140.80 "docker exec cuda_redis redis-cli FLUSHALL"

# Посмотреть логи бота
ssh -i ~/.ssh/cudacrimea_key root@5.83.140.80 "docker logs cuda_bot --tail 50"
```
