# cudaCrimea → SaaS-платформа: Дорожная карта

> **Этот документ — главная карта проекта.** Claude Code читает её первой при каждой сессии работы. Хозяин проекта читает её, когда возвращается после перерыва.
>
> **Дата создания:** 26 мая 2026
> **Цель:** Превратить существующий проект cudaCrimea в SaaS-платформу для региональных админов пабликов.

---

## 📌 БЫСТРЫЙ КОНТЕКСТ (читать первым)

### Кто работает
- **Хозяин проекта (далее "владелец"):** военнослужащий, технический склад ума, понимает логику кода, но сам не пишет. Время работы: ~1 час в день, возможны пропуски 1-2 недели из-за командировок.
- **Claude Code:** делает всю кодовую работу. Получает задачи от владельца, выполняет, объясняет ЧТО и ЗАЧЕМ простым языком.
- **Веб-Claude (этот документ):** стратег. Помогает владельцу формулировать задачи, разбирать решения, держать общую картину.

### Что есть сейчас
- Существующий код агрегатора событий Крыма (~18000 строк Python, FastAPI + aiogram + PostgreSQL + Redis в Docker)
- Репозиторий: https://github.com/asex9876/cudaCrimea
- AI-парсинг событий через LLM (классификатор + экстрактор)
- Админка с UGC-модерацией, парсерами, статистикой
- Документация в `docs/` и `ТЗ/`
- Проект сейчас **не запущен**. Хостинг кончился, бот удалён.

### Куда идём (большое видение)
Не "ещё один агрегатор событий", а **платформа активного отдыха**:
1. Человек открывает (бот/сайт) и видит, что происходит вокруг него
2. Может забронировать место/билет
3. Может вызвать такси до места
4. Может найти попутчиков
5. Может получать персональные рекомендации

Это видение цели. Реализуется поэтапно, годами. Не пытаемся всё сразу.

### Бизнес-модель
**B2B SaaS:** продаём готовый бот/платформу региональным админам пабликов.
- Целевой клиент: админ паблика "Подслушано в [город]" с аудиторией 50-500 тыс.
- Что получает клиент: готовый бот для своего региона + админка для управления
- Что платит: подписка 5-15 тыс. руб/мес (зависит от тарифа)
- Как масштабируется: один код — много клиентов (мультитенантная архитектура)

## 🎯 ПРИНЦИПЫ РАБОТЫ

### Главные правила
1. **Работающее лучше идеального.** Запускаем как только базовая версия работает, чиним по ходу.
2. **Один шаг за раз.** Не пытаемся делать пять задач параллельно. Закончили одну — взялись за следующую.
3. **Каждая задача оставляет код в рабочем состоянии.** После любого изменения проект должен запускаться. Не оставляем "недостроенное" между сессиями.
4. **Документация важна.** Каждое нетривиальное изменение фиксируем в комментариях или в этом документе.
5. **Не переписывать то, что работает.** Только если есть конкретная причина.

### Для Claude Code (важно при работе)
- **Перед началом задачи:** прочитать AI_CONTEXT.md и текущую секцию этого документа.
- **Объяснять простым языком:** владелец понимает логику, но не код. Описывать ЧТО ты делаешь и ЗАЧЕМ.
- **Тестировать после изменений:** запускать тесты, проверять что docker-compose поднимается.
- **Коммитить часто:** после каждой завершённой задачи делать коммит с понятным сообщением.
- **Не выдумывать функционал:** если задача не описана в текущей фазе — спросить у владельца.

## 📊 АНАЛИЗ ТЕКУЩЕГО СОСТОЯНИЯ

### Что работает хорошо
- Архитектура (FastAPI + aiogram + Postgres + Redis в Docker)
- AI-парсинг с двухэтапной обработкой
- Структура базы данных (events, placements, llm_usage и т.д.)
- Админка с основными разделами
- Документация и план развития в `ТЗ/`

### Что критично сломано или недоделано (Phase 1 закроет это)
1. **USER_CITY хранится в памяти Python** (`app/bot/context.py`). При рестарте бота все настройки пользователей теряются.
2. **FSM на MemoryStorage** (`app/bot/main.py`). Недозаполненные формы UGC слетают при рестарте.
3. **Дубли папок** `app/bot/middleware/` и `app/bot/middlewares/` (один и тот же файл `album.py`).
4. **Пароли в docker-compose.yml** в открытом виде. Должны быть в `.env`.
5. **Telethon-сессия не настроена** — Telegram-парсер не работает без неё.
6. **Бот удалён** в BotFather, нужно создавать заново.
7. **Хостинг кончился**, нужно перенастраивать с нуля.

### Что не доделано (Phase 2-3 закроют)
- Настройки пользователя (уведомления, любимые категории)
- Уведомления о новых событиях
- Социальный слой (кто идёт, комментарии)
- Сохранённые события и напоминания
- Расширение на другие города кроме Крыма

### Что нужно построить заново (Phase 4-5)
- Мультитенантная архитектура (для SaaS)
- Личный кабинет клиента-админа
- Биллинг и подписки
- Автоматический онбординг новых клиентов
- Веб-сайт как альтернативный канал доступа

## 🧠 КАК ПРОДУМЫВАТЬ ЛОГИКУ ФУНКЦИЙ (важно!)

### Правило
Перед каждой нетривиальной задачей делаем **мини-ТЗ** — короткий документ с продуманной логикой. Это нужно, чтобы Claude Code не принимал решения за владельца. Без мини-ТЗ Claude Code сделает "как принято", а это может не совпадать с тем, что нужно.

### Какие задачи требуют мини-ТЗ
- Любая новая фича для пользователя (уведомления, избранное, социальный слой)
- Любая фича для админа-клиента (личный кабинет, биллинг)
- Изменения в логике поиска или ранжирования событий
- Интеграции с внешними сервисами (платежи, такси, бронирования)

### Какие задачи НЕ требуют мини-ТЗ
- Чисто технические правки (перенос данных, рефакторинг, баги)
- Однозначные задачи без вариантов реализации
- Косметика (тексты, иконки, цвета)

### Как делать мини-ТЗ
1. Владелец открывает диалог с веб-Claude (стратегом)
2. Описывает задачу из роадмапа в свободной форме
3. Веб-Claude задаёт уточняющие вопросы — что должно происходить в разных случаях
4. По итогам обсуждения веб-Claude формирует короткий документ (1-2 страницы)
5. Этот документ владелец отдаёт Claude Code как инструкцию

### Шаблон мини-ТЗ

```markdown
# Мини-ТЗ: [название задачи]

## Что делаем
[1-2 предложения о сути задачи]

## Зачем это нужно
[Какую проблему пользователя решает]

## Пользовательский сценарий
1. Пользователь делает X
2. Бот реагирует Y
3. Пользователь видит Z
4. ...

## Логика работы
- При условии A — делать B
- При условии C — делать D
- В крайних случаях E — делать F

## Что НЕ делаем (важно!)
- [Что специально оставляем за рамками]

## Технические детали
- Где хранятся данные
- Какие новые таблицы/поля
- Какие новые эндпоинты API

## Критерий готовности
[Когда считаем задачу сделанной]
```

### Пример (для иллюстрации)
Задача из роадмапа: "Уведомления о новых событиях по интересам"

Без мини-ТЗ Claude Code напишет "что-то", и потом обнаружится:
- Уведомления приходят каждые 5 минут — спам
- Нельзя отписаться от конкретной категории
- При неактивности пользователя месяц — продолжают слать
- Уведомление приходит в 3 часа ночи

С мини-ТЗ всё это обсуждается ЗАРАНЕЕ:
- Дайджест раз в день в 10:00 утра по времени пользователя
- Можно настроить любые комбинации категорий и городов
- После 30 дней неактивности — пауза до возврата пользователя
- Если событий меньше 3 — не шлём дайджест в этот день

## 🚨 ЛОВУШКИ, КОТОРЫХ ИЗБЕГАЕМ

### Технические ловушки
1. **Не переписывать рабочий код.** Если что-то работает — оставляем. Меняем только то, что сломано или мешает.
2. **Не добавлять зависимости без необходимости.** Каждая новая библиотека = риск.
3. **Не делать "большой рефакторинг" без чёткой цели.** Только точечные изменения с понятной целью.
4. **Тесты пишем для критичных мест.** Не всё подряд, но базовые сценарии (старт бота, поиск событий, парсинг) — обязательно.
5. **Не уходить в идеальную архитектуру.** SOLID, DDD, чистая архитектура — это хорошо, но не цель. Цель — работающий продукт.

### Продуктовые ловушки
1. **Не добавлять фичу "потому что прикольно".** Каждая новая фича должна решать проблему пользователя.
2. **Не пытаться угодить всем.** Узкая целевая аудитория > широкая.
3. **Не делать функционал, который не будут продавать клиенту.** Если фича не помогает админу-клиенту удерживать своих пользователей — она не нужна.

### Психологические ловушки (для владельца)
1. **Не пытаться сделать всё за один заход.** Час в день = один маленький шаг.
2. **Пропуск 1-2 недели = норма.** Не повод бросать. Возвращаемся к документу, смотрим где остановились.
3. **Сравнение со "взрослыми" SaaS-продуктами демотивирует.** Они начинались так же.
4. **Не зависать в идеях.** Если задумка не реализуется за неделю — упрощать или откладывать.

## 📋 ШАБЛОН СЕССИИ РАБОТЫ

Каждый раз, садясь за проект, владелец делает следующее:

1. Открыть этот документ (SAAS_ROADMAP.md)
2. Посмотреть текущую фазу и активные задачи
3. Выбрать одну задачу (не больше) — ту, что следующая в текущей фазе
4. Дать её Claude Code с контекстом: "Сейчас в фазе X, задача Y. Сделай шаг."
5. Дождаться результата, проверить, что код работает
6. Если работает — отметить задачу как выполненную в этом документе, закоммитить
7. Если что-то непонятно — спросить веб-Claude (этот документ) для разбора

**Если время кончилось посередине задачи** — закоммитить промежуточный результат с пометкой "WIP" (work in progress), в следующий раз продолжить.

## 📞 ПРАВИЛА КОММУНИКАЦИИ С CLAUDE CODE

### Хорошие промпты (пример)
> "Прочти SAAS_ROADMAP.md, сейчас Фаза 1, задача 1.1. Перенеси USER_CITY в БД согласно описанию. После выполнения объясни простым языком что сделал."

### Плохие промпты (избегать)
> "Сделай всё в фазе 1." — слишком большой объём за раз.
> "Допили бот." — непонятно что именно.
> "Сделай как лучше." — Claude Code будет принимать решения, которые потом не понять.

### Когда что-то идёт не так
- Claude Code сделал не то — попросить откатить, объяснить чётче.
- Claude Code не понимает контекст — попросить сначала прочитать AI_CONTEXT.md и SAAS_ROADMAP.md.
- Что-то сломалось после изменений — попросить откатить последний коммит, разобраться.
